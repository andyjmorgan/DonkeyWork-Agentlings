# Go Rewrite Viability Report

**Date:** 2026-03-29
**Status:** Viable — all critical SDKs exist and are production-ready

---

## Executive Summary

A Go rewrite of Agentlings is **fully viable**. Official, actively maintained Go SDKs exist for all three critical dependencies: the Anthropic Messages API, A2A protocol, and MCP protocol. The Go ecosystem is arguably *more mature* than Python for A2A (v2.0.1 stable vs Python's v1.0.0-alpha). The estimated effort is 4,000–5,500 lines of Go code replacing ~1,600 lines of Python.

## Advantages of Go Rewrite

1. **Single static binary** — No Python runtime, no virtualenvs, no pip. Container images shrink from ~200MB+ to ~15–20MB (scratch/distroless).
2. **Lower memory footprint** — Go's runtime uses significantly less memory than CPython.
3. **Native concurrency** — Goroutines are lighter than Python's async tasks; no GIL.
4. **Faster startup** — Near-instant vs Python interpreter startup + module imports.
5. **Built-in tool runner** — The Anthropic Go SDK's `BetaToolRunner` could eliminate the manual tool loop in `loop.py`.
6. **Type safety** — Compile-time errors catch issues that Python discovers at runtime.
7. **Simpler deployment** — `COPY binary /app` in Dockerfile vs multi-stage Python builds.

## Risks and Considerations

### Low Risk
- **SDK availability** — All three SDKs exist and are production-ready.
- **HTTP server** — Go's `net/http` is mature; mounting A2A + MCP on one port is straightforward.
- **Tool implementations** — `os/exec`, `os`, `filepath` map directly.
- **JSONL storage** — File I/O and JSON marshaling are trivial in Go.

### Medium Risk
- **Dual-protocol wiring** — Mounting both A2A (JSON-RPC at `/a2a`) and MCP (Streamable HTTP at `/mcp`) on a single `net/http` server requires careful handler composition. Both SDKs provide HTTP handlers, but ensuring they coexist cleanly needs testing.
- **`compact_20260112` in Go** — The feature works but lives under `Beta` prefix. API surface may shift before going GA.
- **File locking** — `syscall.Flock()` is Unix-only. If Windows support matters, need `sync.Mutex` fallback or explicit Unix-only target.

### Low-Medium Risk
- **More verbose code** — Go requires more boilerplate than Python (error handling, type assertions for `map[string]any`). The 3–4x LOC increase is typical for Python→Go ports.
- **Testing patterns** — Need to replicate the mock LLM client and Docker-based integration test harness. Go's `httptest` package helps for unit tests.

## Recommended Go Dependencies

```go
// go.mod
require (
    github.com/anthropics/anthropic-sdk-go   v1.27.1  // Anthropic Messages API
    github.com/a2aproject/a2a-go/v2          v2.0.1   // A2A protocol
    github.com/modelcontextprotocol/go-sdk   v1.4.1   // MCP protocol
    github.com/kelseyhightower/envconfig     v1.4.0   // Env var config
    gopkg.in/yaml.v3                         v3.0.1   // YAML parsing
)
```

No HTTP framework needed — `net/http` with a simple mux is sufficient.

## Conclusion

**Verdict: Go rewrite is viable and recommended if the goals are smaller binaries, lower resource usage, and simpler deployment.**

All critical SDKs are available, actively maintained, and feature-complete for the Agentlings use case. The A2A Go SDK is actually more mature than its Python counterpart. The main cost is engineering time (~4,000–6,000 LOC) and the inherent verbosity of Go, offset by stronger type safety and dramatically simpler deployment.

The highest-value approach would be to start with the core message loop + Anthropic SDK integration, then layer on A2A and MCP protocol support, mirroring the existing Python architecture.
